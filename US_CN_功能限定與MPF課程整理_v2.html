美國及中國功能限定與 MPF 專利課程整理

課程講師: 盧專利師

課程日期: 2025年3月5日

課程主題: 美國與中國專利中的功能性用語與 Means-Plus-Function (MPF) 規則

課程概述

本課程聚焦於美國與中國專利審查中對功能性限定(Functional Limitation)與手段功能用語(Means-Plus-Function, MPF)的處理方式。由於美國在此領域的法律體系較為完整,課程先探討美國規則,再對照中國的相關規定。

第一部分:美國專利中的功能性用語與 MPF

1.1 功能性用語 vs 結構性限定

基本定義

功能性用語(Functional Language):
- 描述元件「做什麼」(what it does)
- 例如:configured to、adapted to、operable to + 功能描述
- 文義範圍:涵蓋所有能實現所記載功能的裝置

結構性限定(Structural Limitation):
- 描述元件「是什麼」(what it is)
- 明確限定特定結構
- 文義範圍:限於所記載的結構本身

重要原則

物的範圍是涵蓋這個物品是什麼,不是某個物品做什麼。

對於電腦實現的功能性發明,範圍應解釋為硬體與軟體的組合:
- 硬體:CPU、處理器等
- 軟體:執行功能所需的演算法

1.2 MPF 的立法背景與目的

立法歷史

  • 1952年立法:MPF 規則加入美國專利法
  • 法源依據:基於 1946 年最高法院判決見解
  • 立法目的:避免純功能性特徵阻礙創新

立法理由(關鍵引文)

請求項宣告做某件事的所有手段,就是記載它的功能。如果範圍涵蓋實現這個功能的所有手段,將會:
- 阻礙合理的迴避設計
- 給予發明人從未考慮過的結構與方法的排他權
- 抑制解決問題的非均等技術方案

因此,MPF 規則將功能性特徵的範圍限縮至說明書對應結構及其均等物,以平衡專利保護與創新自由。

1.3 35 USC §112(f) - MPF 規則核心

法條要旨

允許以功能描述元件,但保護範圍限縮至:
- 說明書揭露的對應結構(corresponding structure)
- 及其均等物(and equivalents thereof)

MPF 觸發的三要件

  1. 使用 means 或 step 關鍵字
  2. 未明確描述足夠的結構
  3. 以功能方式描述元件

1.4 MPF 判斷的推定原則(Presumption)

使用 means 時

  • 推定為 MPF
  • 舉證責任:申請人需舉證推翻(證明已描述足夠結構)
  • 推翻方式:證明記載了足以實現功能的充分結構

未使用 means 時

  • 推定非 MPF
  • 舉證責任:USPTO 需舉證符合三要件
  • 須證明:(1)使用通用字眼 (2)未記載結構 (3)純功能描述

1.5 Single Means Claim(單一手段請求項)

定義

請求項僅包含單一元件,且該元件為 MPF 用語:

例:A device comprising a means for [功能]

特殊規則

  • 不適用 §112(f): 因為請求項非「組合式請求項」(combination claim)
  • §112(f) 定義:針對組合式請求項中的一個元件(element)
  • 範圍解釋:涵蓋所有能實現該功能的結構(非限於說明書記載)

實務問題

  • Enablement 問題:說明書無法記載所有實現方式
    • 假設實現功能的結構有 100 種
    • 說明書僅記載 50 種
    • 未記載的 50 種導致不滿足充分揭露(Enablement)

電腦實現發明的例外

包含一個處理器執行多個功能的請求項不是 Single Means Claim:

例:A processor configured to:
- execute function A,
- execute function B,
- execute function C

  • 每個程式化的功能視為一個獨立元件
  • 各別功能可能被視為 MPF

1.6 關鍵判例發展

Williamson v. Citrix (2015)

判決要點:
- 確立 module、unit、mechanism 等通用術語可能觸發 MPF
- 關鍵判斷:是否為 nonce word(空泛字眼)+ 純功能描述
- 影響:擴大 MPF 適用範圍

判斷標準:

整個記載是否記載了足以實踐所記載功能的結構

Intel v. Qualcomm (2020)

判決要點:
- 修正 Williamson 過度擴張的趨勢
- 強調:即使使用通用術語,若有結構性描述仍可避免 MPF

實務建議:
描述元件時需包含結構特徵,不能僅用通用詞 + 功能

1.7 如何判斷是否有「結構特徵」

USPTO 判斷標準(來自 2024年3月 Memorandum)

判斷該用語是否具有足以達成功能的結構,可從以下來源確認:

  1. 說明書支持(最重要)

    • 說明書是否明確記載該用語具有結構特徵
    • 建議做法:說明書必須寫清楚

  2. 字典或領域字典

    • 相對不易,字典不一定寫得清楚
    • 不一定能證明結構特徵

  3. 現行技術(Prior Art)

    • 引用 In re Katz 判例(已被選為先例)
    • 現行技術提供證據顯示該用語具有結構特徵
    • 相對較難舉證

實務重點:最可靠的方式是在說明書中明確記載結構

1.8 電腦實現發明的 MPF 要求

軟體結構的定義

根據判例,軟體的結構可以是:
- 演算法(Algorithm)
- 流程圖(Flowchart)
- 特定指令組或規則組

滿足 MPF 的方法

基本原則:
- 通用電腦的基本功能(BIOS 層級):接收、處理、儲存、輸出資料
- 演算法的作用:填補從基本功能到請求項功能的 GAP

公式:

通用電腦基本功能 + 演算法 = 請求項功能

撰寫建議:
- 說明書需記載從基本功能往上加的演算法、公式或流程
- 使演算法與請求項功能建立明確對應關係

1.9 USPTO 2024年3月 Memorandum 重點

備忘錄目的

提醒審查員:
- 應用既有的 MPF 審查指引
- 確保審查記錄的清晰性

審查員應執行事項

  1. 確定特徵是否屬於 §112(f)

    • 明確認定並記錄於審查意見

  2. 記錄清楚

    • 說明為何認定為 §112(f)
    • 指出說明書對應的結構是什麼

  3. 評估說明書支持

    • 檢查是否滿足 §112(a) written description
    • 檢查是否滿足 §112(b) definiteness

Step-Plus-Function (SPF)

  • SPF 通常以 step for 開始
  • step 或 step of 不是 SPF
  • 方法請求項通常以動名詞開始,原則上不是 SPF
  • 實務觀察:大部分是 MPF,很少看到 SPF

1.10 實務案例:過濾元件

案例背景

  • 請求項:過濾元件(filtering element)
  • 說明書:過濾元件(filtering element)
  • 審查員意見:
    • 認定為 MPF
    • 說明書未明確記載對應結構
    • 但從圖式看出該元件就是過濾器(filter)
    • 審查員自行認定圖式結構為對應結構,因此「解套」

風險提示

這個做法存在蠻大的風險

  • 依賴審查員主動解釋
  • 不確定性高
  • 建議:說明書應明確記載對應結構

1.11 收到 MPF 核駁時的答辯策略

Step 1: 爭執 MPF 適用性

主張:
- 已記載足夠結構描述
- 引用 Intel v. Qualcomm 等判例
- 說明非 nonce word(非空泛字眼)

Step 2: 修正權利範圍

考量點:是否應修正請求項?

範圍比較分析

案例:播放裝置
- 說明書對應結構:揚聲器(speaker)
- 審查意見:播放裝置是 §112(f)

方案 文義範圍 均等範圍 審查歷史禁反言
不修正(維持 MPF) 揚聲器及其均等物 有 無
修正為揚聲器 僅揚聲器 可能被阻絕 是(觸發)
修正的風險

修正會觸發審查歷史禁反言(Prosecution History Estoppel):
- 有可能阻絕均等範圍
- 前提:修正是為了克服核駁理由
- 實務困境:
- MPF 核駁通常伴隨 §102/§103 核駁
- 很難主張修正「不是為了克服核駁」
- 幾乎必然成立禁反言

結論

不建議修正請求項:
- 維持 MPF → 保有「對應結構 + 均等物」範圍
- 修正 → 僅剩文義範圍,失去均等保護

Step 3: 說明書補充(限於無新事項)

  • 透過審查歷程檔案建立對應關係
  • 明確指出說明書段落支持
  • 不能引入申請日後的新事項

1.12 MPF 文義範圍與均等論的關係

MPF 文義範圍

構成要件:
- 功能相同(identical function) ← 請求項已明定
- 手段(結構)實質相同
- 結果實質相同

範圍:說明書對應結構及其均等物

均等論(Doctrine of Equivalents, DOE)

構成要件(三不測試):
- 手段(Way)實質相同
- 功能(Function)實質相同

- 結果(Result)實質相同

1.13 MPF 文義侵權與均等侵權的差異

關鍵判例引文

基於實質不同的手段實現相同的功能而達致相同的結果,就沒有文義侵權。

分析邏輯

狀況:被控侵權品以實質不同的手段實現相同功能與結果

MPF 文義侵權判斷:
- 功能相同 ✓ (請求項限定)
- 手段實質相同 ✗ (實質不同)
- → 文義範圍不成立

均等論侵權判斷:
- Way 實質相同 ✗ (實質不同)
- Function 實質相同 ✓
- Result 實質相同 ✓
- → 均等範圍也不成立 (三者皆須實質相同)

結論

  • MPF 文義範圍與均等論範圍的判斷方式相同
  • 若 MPF 文義範圍未滿足(W 或 R 不符) → 均等論也不會滿足
  • 若被控產品執行不同功能(非實質相同) → 文義與均等皆不成立

1.14 審查歷史禁反言(Prosecution History Estoppel)對 MPF 的影響

基本規則

Disclaimer(放棄聲明):
- 發明人在審查歷程中放棄特定結構
- 該放棄的結構不在「其均等物」範圍內
- 對 MPF 文義範圍的「其均等」有限定效果

1.15 MPF 範圍圖解分析

案例:彈性元件(Elastic Element)

技術特徵階層:

功能性描述:彈性元件(具有彈性功能)
↓
上位結構:彈簧(Spring)
↓
具體實施方式:
- 拉伸彈簧(Tension Spring)
- 壓縮彈簧(Compression Spring)

結構性限定寫法

請求項:彈簧(Spring)

文義範圍:
- 彈簧(上位結構)
- 拉伸彈簧
- 壓縮彈簧

均等範圍:
- 彈簧的均等物(實質相同的 Way, Function, Result)
- 拉伸彈簧的均等物
- 壓縮彈簧的均等物

MPF 寫法

請求項:彈性元件(means for providing elasticity)

文義範圍:
- 說明書對應結構:
- 彈簧(上位結構)
- 拉伸彈簧
- 壓縮彈簧
- 及其均等物(相同 Function + 實質相同 W & R)

問題:「其均等物」是否還有均等論的均等範圍?

1.16 MPF「其均等物」的範圍爭議

問題一:是否有雙重均等?

爭議點:
- MPF 文義範圍已包含「對應結構及其均等物」
- 是否還有「均等論的均等範圍」?

答案:

根據 Warner-Jenkinson 判例(1997):

專利法並未創造 MPF 特徵的『其均等物的均等範圍』

結論:其均等物沒有均等論的均等範圍(打叉)

問題二:時間點差異

公告時 vs 侵權時:
- MPF 文義範圍:公告時已存在的均等結構
- 均等論範圍:侵權時存在的均等結構

爭議場景:
- 公告後才出現的新結構
- 該結構為相同功能 + 實質相同 W & R
- MPF 文義範圍抓不到(公告時不存在)
- 均等論是否能抓回來?

1.17 公告後新技術的均等侵權

錯誤判決(已被推翻)

地院判決(非 MPF 特徵):

非 MPF 特徵的 DOE 僅次於公告後出現的技術

這個判決認為:
- 均等範圍只限於「公告時」到「侵權時」之間出現的技術
- 公告前的技術不含在均等範圍

結論:此判決是錯誤的

正確見解

Festo 判例後續發展:
- 均等範圍基於公告前的均等
- 公告後才出現的相同功能均等結構:
- MPF 無法抓(判斷時間點是公告時)
- 但均等論可以抓(判斷時間點是侵權時)

重要判例(2014):

公告時不存在的結構(即新技術),並不排除均等論的均等侵權

推翻例外規則:
- Foreseeability Exception(可預見性例外)已被推翻
- 在公告時無法預見的結構仍可落入均等範圍

1.18 MPF 與均等論的兩大差異

差異一:時間點(Timing)

項目 MPF 文義範圍 均等論範圍
判斷時間 公告時 侵權時
涵蓋結構 公告時存在的對應結構及其均等 侵權時存在的均等結構

差異二:功能要求(Function)

項目 MPF 文義範圍 均等論範圍
功能要求 Identical Function
(相同功能)		 Substantially Same Function
(實質相同功能)
其他要求 實質相同 W & R 實質相同 W & R

1.19 MPF 不侵權與均等論侵權的關係

重要原則:

MPF 的不侵權結論,如果單純基於缺乏相同功能,並不能排除均等論的侵權

分析:
- MPF 要求:Identical Function
- 均等論要求:Substantially Same Function
- 被控產品可能:
- 不滿足 MPF(功能不identical)
- 但滿足均等論(功能 substantially same + W & R substantially same)

結論:
- MPF 文義不侵權 ≠ 均等論不侵權
- 仍需分別判斷

1.20 範圍差異的實務意義

結構性限定 vs MPF

假設:
- 對應結構只有一個
- 或有共同上位結構
- 且無禁反言(Disclaimer)

結論:兩種寫法範圍應該差不多

對應結構的解釋爭議

不確定點:
- 說明書記載:「彈簧」
- 對應結構是否限於說明書這一顆彈簧?
- 還是可以用「彈簧」的文義範圍去讀?
- 包括公告後出現的新型彈簧?

目前狀態:
- 沒有明確判例確認
- 某些判例提到:

如果被控產品不是說明書對應的結構,還要再判斷其均等物

  • 暗示對應結構可能可以文義解讀

1.21 MPF 對應結構的實務考量

多個對應結構的困境

觀察:
- 如果 MPF 對應結構有兩個以上
- 很難有真正的共同上位結構

實例:壓縮彈簧 vs 拉伸彈簧
- 壓縮彈簧:提供推力
- 拉伸彈簧:提供拉力
- 要達成發明目的:
- 提供推力 → 兩元件之間的關係設計不同
- 提供拉力 → 兩元件之間的關係設計不同
- 結論:難以用單一「彈簧」代表兩個結構
- 必須設計成組件形式
- 一個用拉伸彈簧,另一個用壓縮彈簧

推論:
- MPF 對應結構有兩個以上時
- 其上位通常仍是 MPF(功能性描述)
- 較難寫成純結構性描述

1.22 撰寫策略建議

多重保護策略

建議做法:
- 針對重要技術特徵
- 用不同的方法進行包覆

理由:
- 避免單一寫法的風險
- 精準性與廣度的平衡

Code / Program 寫法

來源:客戶端(徐瑞智總監)建議

優點:
- 先立(In re Katz)見解:

寫 Code、Program 或 Software,基本上不是 nonce word

  • 關鍵:這些詞本身具有結構意涵
  • 不會直接觸發 MPF

寫法:

Software code configured to:
- function A
- function B
- function C

支持:很多判例支持此寫法

1.23 軟體發明的特殊性

重要判例引文

軟體元件就跟機構領域不一樣。軟體碼或應用程式的特定結構,某種程式由它的功能來定義。

判斷標準:
- 決定軟體限制條件是否記載足夠結構時
- 法院仍會檢視程式碼或應用程式所記載的功能
- 來決定所屬技術領域人士是否能理解系統用語充分記載明確結構

實務案例:
- 被告專家證人看了說明書記載
- 表示「知道對應哪些結構」
- 因此滿足 definiteness 要求
- 案件通過明確性審查

1.24 Code/Program 寫法的風險考量

優點

  • 不是 nonce word
  • 有判例支持
  • 客戶傾向使用(避免直接被認定為 MPF)

風險

回歸立法理由的擔憂:

每個功能所對應的結構,如果說明書只寫了演算法,它的範圍仍然很廣,因為你根本不知道哪些演算法在你的範圍裡

潛在問題:
- 類似「使用者識別模組」案例
- 如果寫成「使用者識別 code」
- 可能相對好處理
- 但仍有不確定性

目前狀態:
- 到目前為止的判例支持
- 但需持續關注判例發展

實務建議

  • 客戶傾向不用「module」(直接觸發 MPF)
  • 偏好使用「code」和「program」
  • 至少到目前為止的判例可接受

第二部分:中國專利中的功能性限定

2.1 審查指南基本規定

法源:專利審查指南第二部分第二章 3.2.1

原則立場

盡量避免使用功能或者效果的特徵來限定發明

可使用條件(兩個「或」關係):

  1. 無法用結構特徵限定

    • 某技術特徵無法用結構特徵來限定

  2. 功能限定更恰當

    • 用結構特徵限定,不如用功能特徵來限定更為恰當

且需滿足:
- 功能能通過說明書規定的實驗或操作
- 或通常知識者慣用的手段直接肯定地驗證

結論:原則上不允許使用功能限定

2.2 功能性限定的範圍解釋

審查階段

範圍解釋:

權利要求中用功能限定的技術特徵,其範圍解釋為涵蓋了所有能實現該功能的實施方式

結果:
- 範圍非常廣
- 容易找到對比檔案(prior art)
- 較易被核駁

2.3 充分揭露要求(Written Description Support)

判斷標準

問題:功能性限定的特徵是否得到說明書支持?

判斷方法:
1. 所屬領域技術人員(通常知識者)能夠明瞭該功能
2. 可以用說明書沒有提到的其他方式來完成

不得使用的情況

狀況一:審查員想不到第三種實施方式
- 說明書記載兩種實施方式
- 審查員無法想到第三種
- → 不能用功能限定

狀況二:審查員想到的方式不 work
- 審查員想到某種替代方案
- 套進去後無法解決發明所要解決的問題
- 或無法達到相同效果
- → 不能用功能限定

結論:
- 是否得到說明書支持 → 審查員判斷
- 審查員有很大的裁量權

2.4 說明書撰寫要求

不當寫法

模糊描述:

其他的替代方式也可以適用,可以達到這個功能

問題:
- 這種寫法毫無意義
- 不滿足具體揭露要求

正確寫法

  • 說明書應具體描述實現該功能的結構
  • 描述應該具體,不能含糊
  • 不能用 Single Means Claim 形式(純功能性權利要求)

2.5 最高人民法院司法解釋(2020)

功能性限定特徵的定義

完整定義(2020 年版):

對於結構特徵及相互之間的關係未加限定,僅限定所起的功能或效果,且所屬技術領域的技術人員僅通過閱讀權利要求,即可直接、明確地確定實現該功能或效果的具體實施方式的技術特徵

與 2016 年版差異:
- 新增「對於結構特徵及相互之間的關係未加限定」
- 更明確功能性限定的判斷標準

2.6 充分揭露的法律後果

未公開具體實施方式

法條適用:
- 說明書未公開任何能實現該功能或效果的具體實施方式
- → 違反《專利法》第26條第3款
- 權利要求不符合規定

公開的實施方式不充分

問題:
- 說明書公開的實施方式無法實現該功能
- 或與權利要求的技術方案不對應
- → 同樣違反第26條第3款

2.7 功能性限定的判斷方法

判斷標準

與美國的相似性:

如果整句記載了足以實現該功能的充分結構,那它就不是功能限定

所屬技術領域人士的判斷:
- 僅通過閱讀權利要求
- 能否直接、明確地確定具體實施方式
- 如果可以 → 不是功能性限定
- 如果不可以 → 是功能性限定

2.8 功能性限定特徵的範圍解釋

訴訟階段

司法解釋:

功能性限定特徵的內容,應當包括說明書和附圖中記載的所有實現該功能的具體實施方式,及所屬技術領域普通技術人員能夠想到的等同替代方式

與美國的差異

比較項目 中國 美國(MPF)
範圍基礎 說明書所有實施方式 說明書對應結構
均等物 所屬技術領域人員能想到的 侵權時的均等
時間點 - 公告時(文義範圍)

重點差異:
- 中國:說明書所有具體實施方式為基準
- 美國:說明書對應結構 + 其均等物

2.9 中國與美國的主要差異總結

比較項目 美國(MPF) 中國
關鍵字依賴 強(means/step) 弱(無特定關鍵字)
推定機制 有(基於關鍵字) 無
判斷標準 結構描述充分性 充分揭露 + PHOSITA 理解
範圍解釋 對應結構 + 均等物 所有具體實施方式 + 等同替代
時間考量 有(公告時 vs 侵權時) 無明確時間點區分
審查實務 推定原則明確 審查員裁量權大

2.10 台灣審查基準

與美國的相似性

判斷流程:
1. 對應的結構、材料或動作
2. 執行相同功能?
- 是 → 判斷是否相同或均等 → 文義侵權
3. 未執行相同功能 → 判斷侵權時是否符合
- Way、Function、Result 是否實質相同
- 是 → 均等侵權

結論:
- 台灣審查基準與美國判斷方式基本一致
- 區分文義範圍與均等範圍
- 有時間點考量(公告時 vs 侵權時)

第三部分:實務撰寫建議

3.1 美國申請策略

避免觸發 MPF

關鍵原則:增加結構性描述

不良寫法:

❌ means for processing data
❌ module for processing data

改良寫法:

✓ processor configured to process data
✓ processing circuit configured to process data

最佳寫法(結構 + 組成):

✓ processor including a CPU and memory, configured to process data
✓ processing circuit comprising [具體電路元件], configured to...

說明書撰寫要點

對應關係建立:
1. 每個功能性限定需在說明書找到對應實施方式
2. 明確標示結構元件與功能的關聯
3. 提供足夠實施例支持保護範圍

演算法描述:
- 針對電腦實現的發明
- 必須記載演算法、流程圖或指令組
- 填補從通用電腦基本功能到請求項功能的 GAP

3.2 中國申請策略

說明書要求

必要內容:
- 具體實施例:至少一個完整的實施方式
- 結構描述:避免單純功能性描述
- 實現方式:確保所屬領域技術人員可實現

撰寫原則:
- 說明書詳細描述各種實施方式
- 不能僅寫「其他方式也可適用」
- 每個功能性特徵都需對應具體結構

權利要求撰寫

避免純功能性描述:
- 加入結構性限定
- 或用「更恰當」的功能限定(需論證)
- 確保功能可驗證

3.3 跨國申請注意事項

美國申請重點

  1. 謹慎使用術語

    • 避免:means、module、unit、mechanism
    • 改用:processor、circuit、code、program

  2. 增加結構性描述詞

    • comprising、including
    • 具體元件描述

  3. 說明書詳細揭露

    • 對應結構明確
    • 演算法完整

中國申請重點

  1. 說明書需有具體實施例

    • 不能只寫原理
    • 需要完整實施方式

  2. 避免單純功能性描述

    • 除非絕對必要
    • 且需充分論證

  3. 確保可實現性

    • 所屬領域技術人員能理解
    • 能夠實際實施

歐洲/日本

  • 歐洲:相對寬鬆,但仍需明確性
  • 日本:介於美中之間,重視實施可能性

3.4 多重保護策略範例

獨立項多項撰寫

第一項:結構性限定

1. 一種裝置,包含:
處理電路,包括CPU與記憶體,該處理電路經組態以執行資料處理。

第二項:軟體 code 寫法

2. 一種裝置,包含:
軟體程式碼,經組態以執行資料處理,該資料處理包括...

第三項:功能混合結構

3. 一種裝置,包含:
處理器;及
與該處理器耦接的記憶體,該記憶體儲存指令,當該指令被執行時,使該處理器執行...

3.5 審查答辯要點

美國 MPF 核駁

答辯優先順序:
1. 爭執適用性(不是 MPF)
- 主張已有足夠結構
- 引用判例(Intel v. Qualcomm)
- 非 nonce word

  1. 修正說明 (不建議修正請求項)

    • 透過意見書建立對應關係
    • 指出說明書支持段落

  2. 最後手段:限縮至具體實施例

中國功能限定核駁

常見核駁理由:
1. 權利要求保護範圍不清楚
2. 說明書揭露不充分
3. 無法確定具體技術方案

答辯方向:
1. 引用說明書具體段落
- 證明充分揭露
2. 論證 PHOSITA 可理解並實現
3. 必要時修正為更具體的技術特徵

課程總結

核心要點

  1. 功能性用語本身不違法,問題在於保護範圍的解釋與限制

  2. 美國 MPF 規則:

    • 有明確的推定機制(基於關鍵字)
    • 範圍限縮至說明書對應結構及其均等物
    • 文義範圍與均等範圍有時間點差異

  3. 中國功能性限定:

    • 無關鍵字推定,審查員裁量權大
    • 範圍基於說明書所有具體實施方式
    • 審查階段範圍極廣,容易被核駁

  4. 說明書充分揭露是兩國共同的核心要求

  5. 撰寫策略:

    • 結構性描述是避免 MPF 的關鍵
    • 多重保護策略(不同寫法包覆重要特徵)
    • 說明書必須充分對應權利要求

  6. 審查答辯:

    • 優先爭執適用性
    • 謹慎考慮修正的影響(審查歷史禁反言)
    • 建立說明書與請求項的對應關係

  7. 跨國申請:

    • 理解各國差異
    • 美國重視對應結構
    • 中國重視具體實施方式
    • 均需確保所屬領域技術人員可實現

附錄 B: 中國功能限定小結

審查階段

範圍解釋:
- 功能特徵涵蓋所有能實現該功能的結構
- 容易出現對比文件
- 審查員裁量權大

實務困境:
- 特徵採用上位方式撰寫
- 審查階段很可能被要求修正為下位實施例
- 說明書必須記載具體實施方式

訴訟階段

範圍解釋:
- 要求相同功能與相同效果
- 範圍遠小於審查時的範圍
- 小於傳統結構特徵之等同範圍

侵權判斷標準 (2016 司法解釋):
- 以基本相同的手段
- 實現相同的功能
- 達到相同的效果

建議

就中國實務而言,似以結構特徵撰寫為宜

原因:
1. 審查階段範圍過廣,易被核駁
2. 訴訟階段範圍過窄,保護不足
3. 不如直接以結構特徵撰寫,範圍明確
